# 見守り・生存確認機能の導入可否に関する調査レポート

> 調査日: 2026-03-02
> 背景: 行政書士からのフィードバック — 施設に入らず一人暮らしの高齢者に対し、親族が遠方から生存確認できる機能の要望

## 結論: 導入すべき（ただし段階的に）

この機能は**フィーチャークリープではなく、プロダクトの自然な拡張**である。同じユーザーペルソナ、同じ技術基盤、同じ家族連携インフラを活用でき、エンディングノートの「完成したら使わなくなる」という根本的なリテンション課題を解決する。

---

## 1. なぜこの機能がこのアプリに適しているか

### プロダクトとしての整合性

| 観点 | 評価 |
|------|------|
| ユーザーペルソナの一致 | **完全一致** — 一人暮らしの高齢者・遠方の親族 |
| 既存インフラの再利用 | **高い** — 音声AI・家族連携・通知・ライフサイクル管理 |
| 競合との差別化 | **強い** — 音声エンディングノート×音声見守りの組み合わせは存在しない |
| リテンション課題の解決 | **根本的解決** — 単発利用→毎日利用へ転換 |
| ビジネスモデルとの整合 | **良好** — 家族課金の自然な根拠になる |

### エンディングノートとの相乗効果

- エンディングノートは「完成」すると利用頻度が落ちる → 毎日の見守り会話がDAUを維持
- 見守り会話中に「最近お体の調子はどうですか？」→ 自然にノートの医療・介護カテゴリ更新を促せる
- 見守りが機能していれば、万が一の際にライフサイクル管理（死亡報告→同意収集→開封）が迅速に作動する
- **認知機能の変化を検知した場合**、ノートの重要セクション確認を促せる（本人の意思能力があるうちに）

## 2. 市場の状況

### 数字で見る需要

- 65歳以上の一人暮らし世帯: **903万世帯**（全高齢者世帯の52.5%）
- 孤独死: 年間**76,020人**（2024年初の全国集計）、うち8日以上発見されなかった人が21,856人
- 見守りサービス市場規模: **2,270億円**（2025年）→ **3,810億円**（2030年予測）
- 介護家族の48.6%が見守りサービスに関心あり、しかし**実際の利用率はわずか5.0%**

### 利用率が低い理由（＝チャンスでもある）

1. 「どんなサービスかわからない」(31.0%) → **既存アプリ内の追加機能なら認知障壁なし**
2. 「費用負担」(28.2%) → **既存サブスクに含めれば追加コスト感が薄い**
3. 「監視されている感じが嫌」→ **「毎日のおしゃべり」として設計すれば解消**

### AI音声見守りの潮流（2024-2026年）

| サービス | 特徴 | 実績 |
|---------|------|------|
| Clova CareCall | AIが週2回電話、過去の会話を記憶 | 韓国70+自治体導入、応答率96% |
| MaMo. | ChatGPT+ロボット、認知症予防+見守り | 2025年万博出展 |
| Care-Call.ai | AI毎日電話 | 2024年12月ローンチ |
| NTTドコモ「ちかく」| TV接続+ビデオ通話+生活リズム | 月額1,980円 |

## 4. 技術的実現可能性

### 既存コードベースで活用できるもの

| 既存資産 | 見守り機能での活用方法 |
|---------|-------------------|
| WebRTC音声会話基盤 | 短時間の見守り会話セッション |
| AIキャラクター・プロンプト | 同じキャラクターが見守り会話を実施（信頼関係の継続） |
| `familyMembers`テーブル | 通知先の家族関係がすでに定義済み |
| `notifications`テーブル・仕組み | 見守りアラートの配信基盤 |
| `conversations`テーブル | 見守り会話の記録・サマリー保存 |
| `summarizer.ts` | 会話内容から健康状態シグナル抽出 |
| `session-limits.ts` | 見守りセッション用のパラメータ追加 |
| `activityLog` | 監査証跡 |

### 新規に必要なもの

- `wellness_settings`テーブル（見守り設定: 時間帯、頻度、エスカレーション設定）
- `wellness_checkins`テーブル（チェックイン記録: 応答有無、会話要約、健康シグナル）
- Push通知 or SMS送信基盤（Twilio等）— Webアプリの最大の技術課題
- スケジューラー（定期実行: node-cron等）
- 見守り専用の短縮プロンプト

### 最大の技術課題: 通知の到達性

Web Pushの制約:
- iOS SafariはPWAをホーム画面追加した場合のみ対応（16.4+）
- 高齢者にPWA追加を期待するのは非現実的

**推奨**: SMS通知をプライマリ、Web Pushをセカンダリとする。将来的にはアウトバウンドAI電話も検討。

## 5. 倫理・法的考慮事項

### 法規制

- 会話ベースの安否確認に**特別な許認可は不要**
- 個人情報保護法に基づく**適切な同意取得**が必要（家族への情報共有について）
- 緊急通報サービスの場合は警備業法の対象だが、**家族通知のみなら対象外**
- 孤独・孤立対策推進法（2024年4月施行）が**政策的な追い風**

### 倫理的設計原則

1. **「毎日のおしゃべり」であって「監視」ではない** — UX上の言葉遣いが極めて重要
2. **本人起点・本人管理** — 高齢者本人がオプトイン、家族がリモートで有効化できない
3. **透明性** — 家族に何がいつ共有されるか本人が常に確認できる
4. **容易な離脱** — いつでも一時停止・無効化可能（家族の承認不要）
5. **動的同意** — AIが定期的に「お話を続けて大丈夫ですか？」と確認

### 最大のリスク: 責任問題

見守りサービスとして打ち出すと、**アラート未送信時の責任**が発生しうる。
→ **「家族とのつながり機能」としてフレーミング**し、「安全保障サービス」としてはポジショニングしない。利用規約で免責を明記。

## 6. 推奨する段階的実装ロードマップ

### Phase 1: MVP — パッシブ見守り（実装コスト: 小）

- 既存の会話セッション利用状況を家族ダッシュボードに表示
- 「X日間会話がありません」アラートを家族に通知
- **新規開発は最小限**: 通知ロジック + 家族向け表示のみ
- **技術的に最も安全**: 既存の会話行動を活用、新規音声セッション不要

### Phase 2: アクティブ見守り — 毎日の短い会話（実装コスト: 中）

- 毎日決まった時間にPush/SMS通知 →「今日もおしゃべりしませんか？」
- 1-3分の短い音声会話（体調・気分・昨日の出来事）
- 応答/未応答のトラッキング + エスカレーション（1回未応答=何もしない、2回=家族通知、3回=緊急通知）
- 会話内容の要約を家族に共有（本人同意の範囲で）

### Phase 3: インテリジェント見守り（実装コスト: 大）

- 音声バイオマーカー分析（認知機能変化の長期追跡）
- 気分・エネルギーのトレンドダッシュボード
- 認知機能低下検知時にエンディングノートの重要セクション確認を促す
- ヘルスケアプロバイダーへの通知オプション

## 7. ビジネスモデルへの影響

### リテンション改善

- エンディングノート単体: 完成後のDAU低下が予想される
- 見守り機能追加: **単発利用→毎日利用**への転換
- 家族側もダッシュボードを毎日確認 → **ユーザーベースが実質2倍**

### 収益モデル

- **高齢者本人: 無料**（エンディングノート + 基本的な見守り）
- **家族: 有料サブスクリプション**（見守りダッシュボード、アラートカスタマイズ、認知トラッキング）
- 価格帯の参考: AssureOkay $4.99/月、つなまも 無料化、ドコモちかく 1,980円/月
- **推奨: 500〜1,000円/月/家族メンバー**

### B2B2C機会

- 自治体（既存の見守り事業の委託先として）
- 保険会社（リスク低減ツールとして）
- 銀行・信託会社（終活サービスの一環として）
- 行政書士・司法書士事務所（フィードバック元であり、自然な流通チャネル）

## 8. 推奨する次のアクション

1. **Phase 1（パッシブ見守り）から着手する** — 最小の開発コストで市場反応を検証
2. **行政書士に再度ヒアリング** — 具体的なユースケースと紹介可能なテストユーザーの有無
3. **利用規約・プライバシーポリシーの更新** — 見守り機能に関する同意取得フローの設計
4. **SMS通知基盤の調査** — Twilio等の日本国内SMS送信サービスの選定
5. **Phase 2の詳細設計** — 見守り会話のプロンプト設計、エスカレーションロジック
